1 处理事件#

本主题描述JavaFX应用程序中的事件和事件处理。 了解事件类型，事件目标，事件捕获，事件冒泡以及事件处理系统的基础体系结构。

事件用于将用户采取的操作通知您的应用程序，并使应用程序能够响应事件。 JavaFX平台提供了用于捕获事件，将事件路由到其目标以及使应用程序根据需要处理事件的结构。

事件#

事件表示应用程序感兴趣的事件的发生，例如鼠标被移动或按键被按下。 在JavaFX中，事件是 javafx.event.Event 类或 Event的任何子类的实例。 JavaFX提供了几个事件，包括 DragEvent, KeyEvent, MouseEvent, ScrollEvent 等。 您可以通过扩展 Event 类来定义自己的事件。

每个事件都包含 Table 1-1 中描述的信息。

***Table 1-1 Event Properties***
PropertyDescription
Event type发生的事件的类型。
Source事件的起源，关于事件在事件分发链中的位置。 当事件沿链传递时，源发生变化。
Target在其上发生操作的节点以及事件分发链中的结束节点。 目标不会发生变化，但是，如果事件过滤器在事件捕获阶段消耗了事件，则目标将不会接收该事件。

事件子类提供特定于事件类型的其他信息。 例如， MouseEvent 类包含诸如按下哪个按钮，按下按钮的次数以及鼠标的位置之类的信息。

事件类型#

事件类型是 EventType 类的实例。 事件类型进一步将单个事件类的事件分类。 例如， KeyEvent 类包含以下事件类型：

  • KEY_PRESSED
  • KEY_RELEASED
  • KEY_TYPED

事件类型是分层的。 每个事件类型都有一个名称和一个父类型。 例如，被按下的键的事件的名称为KEY_PRESSED，父类型为KeyEvent.ANY。 顶级事件类型的父类型为null。 Figure 1-1 显示了层次结构的子集。

***Figure 1-1 Event Type Hierarchy***

Description of "Figure 1-1 Event Type Hierarchy"

层次结构中的顶级事件类型是Event.ROOT，它等效于Event.ANY。 在子类型中，事件类型ANY用于表示事件类中的任何事件类型。 例如，要对任何类型的键事件提供相同的响应，请使用KeyEvent.ANY作为事件过滤器或事件处理程序的事件类型。 若要仅在释放键时做出响应，请对过滤器或处理程序使用KeyEvent.KEY_RELEASED事件类型。

事件目标#

事件的目标可以是实现EventTarget 接口的任何类的实例。 buildEventDispatchChain的实现创建了事件分发链，事件必须传播到达目标。

Window，Scene和Node类实现EventTarget接口，并且这些类的子类继承该实现。 因此，用户界面中的大多数元素都定义了它们的调度链，从而使您可以专注于响应事件，而不必担心创建事件调度链。

如果创建响应用户操作的自定义UI控件，并且该控件是Window，Scene或Node的子类，则该控件是通过继承的事件目标。 如果控件或控件的元素不是Window，Scene或Node的子类，则必须为该控件或元素实现EventTarget接口。 例如，MenuBar控件是通过继承实现的目标，但是菜单栏的MenuItem元素必须实现EventTarget接口，以便它可以接收事件。

事件传递过程#

事件传递过程包含以下步骤：

  1. 目标选择
  2. 路由构造
  3. 事件捕捉
  4. 事件冒泡

目标选择#

发生操作时，系统会根据内部规则确定哪个节点是目标：

  • 对于按键事件，目标是具有焦点的节点。
  • 对于鼠标事件，目标是光标所在位置的节点。 对于合成的鼠标事件，触摸点被视为光标的位置。
  • 对于由触摸屏上的手势生成的连续手势事件，目标是手势开始时所有触摸的中心点处的节点。 对于由触摸屏以外的其他物体（例如触控板）上的手势生成的间接手势事件，目标是光标位置处的节点。
  • 对于由触摸屏上的滑动生成的滑动事件，目标是所有手指整个路径中心的节点。 对于间接滑动事件，目标是光标所在位置的节点。
  • 对于触摸事件，每个触摸点的默认目标是第一次按下位置处的节点。 可以使用ungrab()，grab()或grab(node)方法为事件过滤器或事件处理程序中的接触点指定其他目标。

如果在光标或触摸处有多个节点，则将最上面的节点视为目标。 例如，如果用户单击或触摸Figure 1-2中所示的三角形，则三角形是目标，而不是包含圆形和三角形的矩形。

***Figure 1-2 Sample User Interface Event Targets***

Description of "Figure 1-2 Sample User Interface Event Targets"

当按下鼠标按钮并选择目标时，所有后续的鼠标事件都会传递到同一目标，直到释放按钮为止。 类似地，对于手势事件，从手势开始到手势完成，手势事件会传递到在手势开始时标识的目标。 触摸事件的默认设置是将事件传递到为每个触摸点确定的初始目标节点，除非使用ungrab()，grab()或grab(node)方法修改了目标。

路由构造#

初始事件路由由在选定事件目标的buildEventDispatchChain()方法的实现中创建的事件分发链确定。 例如，如果用户单击Figure 1-2中所示的三角形，则初始路线将由Figure 1-3中的灰色节点显示。当一个场景图节点被选择作为事件对象，在Node类的buildEventDispatchChain()方法的默认实现中初始事件路由集是从Stage到自身的路径。

可以将路由修改为沿路由处理事件的事件过滤器和事件处理程序。 同样，如果事件过滤器或事件处理程序在任何时候使用了事件，则初始路由上的某些节点可能不会接收到该事件。

事件捕获阶段#

在事件捕获阶段，事件由应用程序的根节点调度，并沿事件调度链向下传递到目标节点。 使用Figure 1-3中所示的事件分发链，事件在事件捕获阶段从Stage节点传播到Triangle节点。

如果链中的任何节点都为发生的事件类型注册了事件过滤器，则将调用该过滤器。 筛选器完成后，事件将传递到链中的下一个节点。 如果未为节点注册筛选器，则事件将传递到链中的下一个节点。 如果没有过滤器消耗该事件，则事件目标最终将接收并处理该事件。

事件冒泡阶段#

在达到事件目标并且所有已注册的筛选器都处理了事件之后，事件将沿着分派链从目标返回到根节点。 使用Figure 1-3中所示的事件分发链，事件在事件冒泡阶段从Triangle节点传播到Stage节点。

如果链中的任何节点都为遇到的事件类型注册了处理程序，则将调用该处理程序。 处理程序完成后，事件将返回到链中的下一个节点。 如果未为节点注册处理程序，则事件将返回到链中的下一个节点。 如果没有处理程序消耗该事件，则根节点最终将接收该事件并完成处理。

事件处理#

事件处理由事件过滤器和事件处理程序提供，它们是EventHandler接口的实现。 如果希望事件发生时通知应用程序，请为该事件注册一个过滤器或一个处理程序。 过滤器和处理程序之间的主要区别在于哪一个何时执行。

事件过滤器#

在事件捕获阶段执行事件过滤器。 父节点的事件过滤器可以为多个子节点提供公共事件处理，如果需要，可以消耗该事件以防止子节点接收该事件。 当事件经过注册过滤器的节点时，将执行针对发生的事件类型注册的过滤器。

一个节点可以注册多个过滤器。 每个过滤器的调用顺序基于事件类型的层次结构。 针对特定事件类型的过滤器将在针对通用事件类型的过滤器之前执行。 例如，MouseEvent.MOUSE_PRESSED事件过滤器将在InputEvent.ANY事件过滤器之前调用。没有指定执行相同级别的两个过滤器的顺序。

事件处理程序#

在事件冒泡阶段执行事件处理程序。 如果子节点的事件处理程序不消耗该事件，则父节点的事件处理程序可以在子节点处理事件后对该事件进行操作，并可以为多个子节点提供公共事件处理。 当事件通过注册处理程序的节点返回时，将执行针对发生的事件类型注册的处理程序。

一个节点可以注册多个处理程序。 调用每个处理程序的顺序基于事件类型的层次结构。 特定事件类型的处理程序在通用事件类型的处理程序之前执行。 例如对于KeyEvent.KEY_TYPED事件的处理程序在InputEvent.ANY事件的处理程序之前被调用。 没有指定执行同一级别的两个处理程序的顺序，但最后执行由使用便捷方法中描述的便捷方法注册的处理程序的情况除外。

消费事件#

事件过滤器或事件处理程序可以在事件分配链中的任何位置通过调用consume()方法来消耗事件。 此方法表示事件处理已完成，事件分发链的遍历结束。

在事件过滤器中消耗该事件可防止事件分发链上的任何子节点被该事件触发。 在事件处理程序中消耗该事件将停止事件分发链上的父处理程序对事件的任何进一步处理。 但是，如果使用该事件的节点为该事件注册了多个过滤器或处理程序，则对等过滤器或处理程序仍将执行。

例如，使用Figure 1-3中所示的事件分发链，假定Pane节点具有为KeyEvent.KEY_PRESSED事件注册的事件过滤器和为InputEvent.ANY事件注册的事件过滤器。 如果按键事件的过滤器消耗了该事件，输入事件的过滤器将执行，并且Triangle节点不会接收到该事件。

请注意，JavaFX UI控件的默认处理程序通常会消耗大多数输入事件。
